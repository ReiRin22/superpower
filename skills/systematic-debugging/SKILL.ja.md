# 体系的デバッグ

## 概要

場当たり的な修正は時間を浪費し、新しいバグを作る。応急パッチは根本問題を隠す。

**中核原則:** 修正を試みる前に、必ず根本原因を見つける。症状修正は失敗である。

**このプロセスの字面に違反することは、デバッグの精神に違反することである。**

## 鉄則

```text
根本原因調査なしに修正してはならない
```

Phase 1 を完了していないなら、修正案を提案できない。

## 使うタイミング

あらゆる技術的問題に使う。

- テスト失敗
- 本番バグ
- 予期しない挙動
- パフォーマンス問題
- ビルド失敗
- 統合問題

**特に使うべき時:**

- 時間的圧力がある (緊急時ほど推測したくなる)
- 「すぐ直せる」が明らかに見える
- すでに複数の修正を試した
- 前の修正が効かなかった
- 問題を完全には理解していない

**飛ばしてはいけない時:**

- 問題が単純に見える (単純なバグにも根本原因がある)
- 急いでいる (急ぐほど手戻りが確実になる)
- 管理者が今すぐ直せと言う (体系的な方が手当たり次第より速い)

## 4 つのフェーズ

各フェーズを完了してから次へ進むこと。

### Phase 1: 根本原因調査

**修正を試みる前に:**

1. **エラーメッセージを注意深く読む**
   - エラーや警告を読み飛ばさない
   - そこに正確な解決策が含まれることが多い
   - スタックトレースを最後まで読む
   - 行番号、ファイルパス、エラーコードを記録する

2. **一貫して再現する**
   - 確実に発生させられるか
   - 正確な手順は何か
   - 毎回起きるか
   - 再現できないなら、推測せず追加データを集める

3. **最近の変更を確認する**
   - 何が変わってこの問題を起こし得るか
   - git diff、最近のコミット
   - 新しい依存関係、設定変更
   - 環境差分

4. **複数コンポーネントのシステムで証拠を集める**

   **CI -> build -> signing、API -> service -> database のように複数コンポーネントがある場合:**

   **修正案の前に診断用 instrumentation を追加する:**

   ```text
   各コンポーネント境界について:
     - コンポーネントへ入るデータをログする
     - コンポーネントから出るデータをログする
     - 環境/設定が伝播しているか確認する
     - 各レイヤーの状態を確認する

   一度実行して、どこで壊れるかを示す証拠を集める
   その証拠を分析して失敗コンポーネントを特定する
   そのコンポーネントを調査する
   ```

   **例:**

   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

5. **データフローを辿る**

   **エラーが深い call stack で起きている場合:**

   完全な逆方向トレース技法は、このディレクトリの `root-cause-tracing.md` を参照する。

   **短縮版:**

   - 悪い値はどこで発生したか
   - 誰がその悪い値で呼び出したか
   - 発生源を見つけるまで上流へ辿る
   - 症状ではなく発生源で直す

### Phase 2: パターン分析

**修正前にパターンを見つける:**

1. **動いている例を探す**
   - 同じコードベース内の似た動作コードを見つける
   - 壊れているものに似ていて、動いているものは何か

2. **参照実装と比較する**
   - パターンを実装するなら、参照実装を完全に読む
   - 流し読みしない。全行読む
   - 適用前にパターンを完全に理解する

3. **差分を特定する**
   - 動くものと壊れたものの違いは何か
   - どれだけ小さくてもすべて列挙する
   - 「これは関係ない」と決めつけない

4. **依存関係を理解する**
   - 他に必要なコンポーネントは何か
   - 設定、config、環境は何か
   - どんな前提を置いているか

### Phase 3: 仮説とテスト

**科学的方法:**

1. **単一の仮説を立てる**
   - 「X が根本原因だと思う。理由は Y」と明確に述べる
   - 書き留める
   - 曖昧にせず具体的にする

2. **最小限にテストする**
   - 仮説検証のための最小変更をする
   - 変数は一度に一つ
   - 複数の修正を同時にしない

3. **続行前に検証する**
   - 効いたか。Yes なら Phase 4
   - 効かなかったか。新しい仮説を立てる
   - 修正を積み重ねない

4. **分からない時**
   - 「X が分からない」と言う
   - 分かったふりをしない
   - 助けを求める
   - さらに調査する

### Phase 4: 実装

**症状ではなく根本原因を直す:**

1. **失敗テストケースを作る**
   - 可能な限り単純な再現
   - 可能なら自動テスト
   - フレームワークがなければ一回限りのテストスクリプト
   - 修正前に必須
   - 適切な失敗テストを書くには `superpowers:test-driven-development` スキルを使う

2. **単一の修正を実装する**
   - 特定した根本原因に対応する
   - 一度に一つの変更
   - 「ついでに」改善しない
   - リファクタリングを同梱しない

3. **修正を検証する**
   - テストは今通るか
   - 他のテストは壊れていないか
   - 問題は実際に解決したか

4. **修正が効かない場合**
   - 停止する
   - 試した修正の数を数える
   - 3 未満なら Phase 1 へ戻り、新情報で再分析する
   - **3 以上なら停止し、アーキテクチャを疑う**
   - アーキテクチャ議論なしに Fix #4 を試みない

5. **3 回以上失敗した場合: アーキテクチャを疑う**

   **アーキテクチャ問題を示すパターン:**

   - 各修正が別の場所の共有状態/結合/問題を露出させる
   - 修正に「大規模リファクタリング」が必要になる
   - 各修正が別の症状を生む

   **停止して根本を問う:**

   - このパターンは根本的に健全か
   - 惰性だけで続けていないか
   - 症状修正を続けるより、アーキテクチャをリファクタすべきか

   追加修正の前に human partner と相談する。

   これは仮説失敗ではない。間違ったアーキテクチャである。

## 危険信号 - 停止してプロセスに従う

以下を考えたら停止する。

- 「今だけ quick fix、後で調査」
- 「X を変えて動くか試そう」
- 「複数変更してテストを走らせよう」
- 「テストは飛ばして手動確認」
- 「たぶん X だから直そう」
- 「完全には理解していないが、これで動くかも」
- 「パターンは X だが、違う形で適応しよう」
- 「主な問題はこれです」と調査なしに修正を列挙する
- データフローを辿る前に解決策を提案する
- **すでに 2 回以上試した後の「もう一回だけ」**
- **各修正が別の場所の新問題を出す**

**これらはすべて Phase 1 へ戻る合図である。**

**3 回以上修正が失敗した場合:** Phase 4.5 に従いアーキテクチャを疑う。

## human partner からの「間違った進め方」シグナル

**以下のリダイレクトに注意する:**

- "Is that not happening?" - 検証せず仮定した
- "Will it show us...?" - 証拠収集を追加すべきだった
- "Stop guessing" - 理解せず修正案を出している
- "Ultrathink this" - 症状ではなく根本を疑うべき
- "We're stuck?" - アプローチが機能していない

**見たら:** 停止して Phase 1 へ戻る。

## よくある合理化

| 言い訳 | 現実 |
|--------|------|
| 「単純だからプロセス不要」 | 単純な問題にも根本原因がある。プロセスは単純バグでも速い。 |
| 「緊急で時間がない」 | 体系的デバッグは推測の連打より速い。 |
| 「先にこれだけ試してから調査」 | 最初の修正が流れを作る。最初から正しくやる。 |
| 「修正が動くのを確認してからテストを書く」 | 未テスト修正は定着しない。テストファーストが証明する。 |
| 「複数修正を同時にすれば速い」 | 何が効いたか分からない。新バグを生む。 |
| 「参照が長いからパターンだけ適応する」 | 部分理解はバグを保証する。全部読む。 |
| 「問題が見えた。直そう」 | 症状を見ることは根本原因理解ではない。 |
| 「もう一回だけ」(2 回以上失敗後) | 3 回以上の失敗 = アーキテクチャ問題。修正ではなくパターンを問う。 |

## クイックリファレンス

| Phase | 主な活動 | 成功基準 |
|------|----------|----------|
| **1. Root Cause** | エラーを読む、再現、変更確認、証拠収集 | WHAT と WHY を理解する |
| **2. Pattern** | 動作例を探し比較する | 差分を特定する |
| **3. Hypothesis** | 仮説を立て、最小検証する | 確認済み、または新仮説 |
| **4. Implementation** | テスト作成、修正、検証 | バグ解決、テスト成功 |

## 「根本原因なし」が分かった場合

体系的調査の結果、本当に環境依存、タイミング依存、外部要因だった場合:

1. プロセスを完了している
2. 調査内容を文書化する
3. 適切な扱いを実装する (retry, timeout, error message)
4. 将来調査のため monitoring/logging を追加する

**ただし:** 「根本原因なし」の 95% は調査不足である。

## 補助技法

このディレクトリには体系的デバッグの一部である技法がある。

- **`root-cause-tracing.md`** - call stack を逆に辿り、元のトリガーを見つける
- **`defense-in-depth.md`** - 根本原因発見後、複数レイヤーに validation を追加する
- **`condition-based-waiting.md`** - 任意 timeout を condition polling に置き換える

**関連スキル:**

- **superpowers:test-driven-development** - 失敗テストケース作成用 (Phase 4, Step 1)
- **superpowers:verification-before-completion** - 成功主張前に修正が効いたことを検証する

## 実世界での効果

デバッグセッションから:

- 体系的アプローチ: 修正まで 15-30 分
- ランダム修正アプローチ: 2-3 時間の迷走
- 初回修正成功率: 95% vs 40%
- 新規バグ導入: ほぼゼロ vs よく起きる
